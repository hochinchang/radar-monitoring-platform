# 需求文件

## 簡介

「電腦即時狀況」頁面目前分為兩個獨立區塊：「系統負載與記憶體」和「磁碟使用狀況」，各自以科別分組顯示卡片。本功能將兩個區塊合併，每台電腦以單一統一卡片呈現，卡片內同時顯示 IP、儀器名稱、記憶體使用率、CPU 負載（1m / 5m / 15m）及該台電腦所有磁碟的使用率，讓操作人員一眼掌握每台電腦的完整狀態。

## 詞彙表

- **統一卡片頁面（UnifiedCard_Page）**：重新設計後的「電腦即時狀況」頁面（`computers.html` + `computers.js`）
- **統一卡片（Unified_Card）**：每台電腦對應的單一卡片，包含系統與磁碟所有指標
- **電腦狀態 API（Computer_Status_API）**：後端新增的合併端點 `GET /api/v1/computers/current`，回傳每台電腦的系統與磁碟合併資料
- **系統服務（System_Service）**：`backend/services/system_service.py` 中的系統狀態查詢邏輯
- **磁碟項目（Disk_Entry）**：單一磁碟掛載點的資料，包含 `file_system`（如 `/`、`/data`）與 `used_pct`
- **科別（Department）**：電腦所屬科別，取值為 `wrs`、`mrs`、`sos`、`dqcs`、`rsa`
- **警示等級（Alert_Level）**：燈號等級，依嚴重程度排序為 `ok` < `yellow` < `orange` < `red`
- **CPU 逾時（CPU_Timeout）**：距上次系統資料更新的時間差（分鐘）

## 需求

### 需求 1：合併後端資料端點

**使用者故事：** 身為前端開發者，我希望透過單一 API 端點取得每台電腦的系統與磁碟合併資料，以便減少前端請求次數並簡化資料整合邏輯。

#### 驗收條件

1. 電腦狀態 API 應提供 `GET /api/v1/computers/current` 端點，回傳包含 `items` 陣列的 JSON 物件。
2. 當端點被呼叫時，電腦狀態 API 應針對每個唯一 IP 位址回傳一筆資料，包含 `ip`、`equipment_name`、`department`、`load_1`、`load_5`、`load_15`、`memory_use`、`server_time` 及 `disks` 陣列。
3. 電腦狀態 API 應將所有符合相同 IP 的磁碟項目填入 `disks` 陣列，每筆包含 `file_system` 與 `used_pct`。
4. 當某台電腦沒有磁碟記錄時，電腦狀態 API 應回傳空的 `disks` 陣列。
5. 若 SystemStatus 資料庫無法連線，電腦狀態 API 應回傳 HTTP 503 並附上描述性錯誤訊息。
6. 若 DiskStatus 資料庫無法連線，電腦狀態 API 應為每台電腦回傳空的 `disks` 陣列，並在回應中將 `disk_error` 旗標設為 `true`。

---

### 需求 2：統一卡片版面

**使用者故事：** 身為操作人員，我希望每台電腦以單一卡片呈現所有指標，以便不需要在兩個區塊之間來回對照同一台電腦的資料。

#### 驗收條件

1. 統一卡片頁面應為每台電腦顯示且僅顯示一張統一卡片，取代原本的雙區塊版面。
2. 渲染統一卡片時，統一卡片頁面應將電腦的 IP 位址顯示為卡片標題。
3. 渲染統一卡片時，統一卡片頁面應在 IP 位址下方顯示儀器名稱。
4. 渲染統一卡片時，統一卡片頁面應以一位小數顯示記憶體使用率百分比（例如：`72.3%`）。
5. 渲染統一卡片時，統一卡片頁面應顯示 1m、5m、15m 的 CPU 負載值，各以兩位小數呈現。
6. 渲染統一卡片時，統一卡片頁面應顯示該電腦的所有磁碟項目，每筆顯示 `file_system` 與一位小數的 `used_pct`（例如：`/ 50.0%`、`/data 30.0%`）。
7. 當指標值不可用（null）時，統一卡片頁面應對該指標顯示 `N/A`。

---

### 需求 3：科別分組

**使用者故事：** 身為操作人員，我希望統一卡片依科別分組顯示，以便快速定位特定科別的電腦狀態。

#### 驗收條件

1. 統一卡片頁面應依科別將統一卡片分組，順序為：`wrs`（氣象雷達科）、`mrs`（海象雷達科）、`sos`（衛星作業科）、`dqcs`（品管科）、`rsa`（應用科）。
2. 當電腦所屬科別不在預定義順序中時，統一卡片頁面應將該群組顯示在所有預定義群組之後。
3. 統一卡片頁面應以中文科別名稱作為每個群組的標題。

---

### 需求 4：警示燈號

**使用者故事：** 身為操作人員，我希望統一卡片以顏色燈號標示異常指標，以便立即識別需要關注的電腦。

#### 驗收條件

1. 當記憶體使用率超過 80% 時，統一卡片頁面應對記憶體指標顯示套用警示等級 `red`。
2. 當記憶體使用率超過 70% 且不超過 80% 時，統一卡片頁面應對記憶體指標顯示套用警示等級 `orange`。
3. 當記憶體使用率超過 60% 且不超過 70% 時，統一卡片頁面應對記憶體指標顯示套用警示等級 `yellow`。
4. 當磁碟項目的 `used_pct` 使得剩餘空間（100 − used_pct）低於 1% 時，統一卡片頁面應對該磁碟項目顯示套用警示等級 `red`。
5. 當磁碟項目的剩餘空間低於 5% 且不低於 1% 時，統一卡片頁面應對該磁碟項目顯示套用警示等級 `orange`。
6. 當磁碟項目的剩餘空間低於 10% 且不低於 5% 時，統一卡片頁面應對該磁碟項目顯示套用警示等級 `yellow`。
7. 當 CPU 逾時超過 30 分鐘時，統一卡片頁面應對卡片邊框套用警示等級 `red`。
8. 當 CPU 逾時超過 10 分鐘且不超過 30 分鐘時，統一卡片頁面應對卡片邊框套用警示等級 `orange`。
9. 當 CPU 逾時超過 3 分鐘且不超過 10 分鐘時，統一卡片頁面應對卡片邊框套用警示等級 `yellow`。
10. 統一卡片頁面應將所有指標（記憶體、所有磁碟、CPU 逾時）中最嚴重的警示等級套用至整張卡片的邊框顏色。

---

### 需求 5：自動刷新與手動刷新

**使用者故事：** 身為操作人員，我希望頁面定期自動更新資料，並可手動觸發刷新，以便隨時掌握最新狀態。

#### 驗收條件

1. 統一卡片頁面應每 60 秒自動從電腦狀態 API 刷新資料。
2. 當刷新按鈕被點擊時，統一卡片頁面應立即從電腦狀態 API 取得最新資料，並重置 60 秒自動刷新計時器。
3. 當資料取得成功時，統一卡片頁面應將「上次更新」時間戳記更新為當前本地時間。
4. 若資料取得因網路錯誤或逾時而失敗，統一卡片頁面應顯示警告訊息，且不清除已渲染的卡片。
5. 若資料取得因伺服器錯誤（HTTP 5xx）而失敗，統一卡片頁面應顯示錯誤訊息，說明資料庫連線失敗。

---

### 需求 6：移除舊版雙區塊版面

**使用者故事：** 身為開發者，我希望移除舊版的「系統負載與記憶體」和「磁碟使用狀況」兩個獨立區塊，以便頁面結構保持簡潔一致。

#### 驗收條件

1. 統一卡片頁面應包含單一 `<section>` 元素（id 為 `computers-container`）以容納所有統一卡片，取代原本的 `system-container` 與 `disk-container` 元素。
2. 統一卡片頁面不得直接從電腦頁面呼叫舊版的 `/api/v1/system/current` 或 `/api/v1/disk/current` 端點。
